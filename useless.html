Core Identity

dew.codes — Kevin's Public Internet OS

Not a portfolio. Not a blog. A network of useful mini-products under a personal brand. Each pillar is a standalone mini-website. Together they form something no one can easily replicate.

TAGLINE
"Useful tools and field notes from a Singapore builder."

Specific, humble, expandable. Every section of the site can stand alone — someone visiting /finance sees a finance research hub. Someone visiting /entertainment/anime sees an anime tracker. The unifying thread is only visible if they poke around.

🔨 Build
/projects · /lab · /writing
Software projects, dev logs, product experiments, open source. Case studies not just project cards.
livelingdiabetes-connectnuschedulelab
📈 Track
/finance
Markets, dashboards, quant tools, sector research. Interactive toys, not dense theory.
uraniumreitsdcabacktester
🌏 Explore
/travel · /singapore
Country guides, diving logbook, field guides, ecology. Practical not pretty.
japandivingfield-guidenus
🎵 Curate
/entertainment
Music, anime, guitar, J-pop, Mandopop. Interactive and playable, not just lists.
jpopguitaranimetier-list
🛠 Tools
/tools (top-level)
The utility engine. Highest traction section. Tools earn backlinks. Blogs don't.
cap-planneranime-pickertrip-budget
💛 Support
/support · /community
Transparent costs, tasteful tiers, guestbook, contributor wall.
donatewallnewsletter
The key principle: Each vertical (/finance, /entertainment/anime, /travel, /singapore) should have its own nav, own hero, own tools, own personality. A stranger who lands on /finance thinks "this is a finance research site by a Singapore student." Only when they notice the global nav do they realise there's a whole world.

Implementation
Next.js App Router — What You Must Know

Everything specific to building dew.codes on Next.js 14+ App Router. Non-negotiable setup items first, then enhancements.

1
App Router folder structure for this architecture
▼

Your routes map directly to folders in app/. Each section gets its own layout.tsx for the standalone mini-site feel.

// app/ directory structure app/ ├── layout.tsx // global: fonts, analytics, cmd+K ├── page.tsx // homepage command center │ ├── (about)/ │ └── about/ │ ├── layout.tsx // about-specific nav/wrapper │ ├── now/page.tsx │ └── stack/page.tsx │ ├── (finance)/ │ └── finance/ │ ├── layout.tsx // finance mini-site layout — own header, disclaimer banner │ ├── page.tsx │ └── quant-tools/ │ └── [slug]/page.tsx │ ├── (travel)/ │ └── travel/ │ ├── layout.tsx // travel mini-site layout — own header, travel palette │ └── countries/ │ └── [country]/ │ ├── layout.tsx // country-specific layout │ └── [section]/page.tsx │ ├── tools/ │ └── [slug]/ │ ├── page.tsx │ └── opengraph-image.tsx // per-tool OG image │ ├── sitemap.ts // auto-generated, include all content ├── robots.ts └── not-found.tsx
Use route groups (finance), (travel), (entertainment) to scope layouts without affecting the URL. Each vertical gets its own layout.tsx with its own nav, palette, and header — this is what makes each section feel like a standalone site.

Universal Content Model

Everything shares a base type. This makes sitemaps, RSS feeds, search indexes, tag pages, related content, and OG images automatic.

1
Base type — DewContent
▼
type DewContent = { // identity title: string slug: string description: string category: 'project' | 'travel' | 'finance' | 'music' | 'anime' | 'tool' | 'writing' | 'singapore' tags: string[] // shared across categories — enables cross-linking // timestamps datePublished: string dateUpdated: string // show prominently on finance/travel/tool pages // status status: 'draft' | 'published' | 'archived' featured: boolean // content readingTime?: number sourceLinks?: string[] // for finance/data pages — cite your sources relatedPages?: string[] // slugs — auto-generate "related" sections canonicalUrl: string }

APIs & External Data — Complete Reference

Every API you'll use, its limits, auth requirements, and how to use it in Next.js.

APIUsed forAuthCostNotes
Jikan v4 /anime, /seasonal, /news None Free Unofficial MAL wrapper. Rate limit: 3 req/sec, 60/min. Cache aggressively. Use /seasons/now for seasonal board.
AniList GraphQL Watching list, scores, studio data OAuth (for personal data) Free Preferred over Jikan for personal list. Use GraphQL to fetch exactly what you need. No rate limit if reasonable.
GitHub REST v3 / GraphQL v4 Contribution heatmap, repos, changelog PAT (personal access token) Free Store PAT in env var. Use contributionsCollection for heatmap. Auto-generate changelogs from releases endpoint.
Spotify Web API Now playing, top tracks, playlists OAuth + PKCE (user auth) Free Keep client_secret server-side in /api/music. Refresh token flow required. See: spotify-now-playing pattern.
Alpha Vantage Stock prices, forex, economic data API key Free (25 req/day) Free tier very limited. Cache responses in KV store (Upstash/Vercel KV). Or use Yahoo Finance unofficial.
FRED API Macro data: interest rates, CPI, GDP API key (free) Free Federal Reserve data. Very reliable. Great for macro dashboard. Register at fred.stlouisfed.org
frankfurter.app Exchange rates None Free Open-source, no API key needed. Daily rates from ECB. Perfect for /travel/tools/exchange-rate-cheatsheet
NUSMods API Module data, timetable, workload None Free Public API at api.nusmods.com. Module codes, MC counts, workload ratings, exam dates. Perfect for /singapore/student-tools.
iNaturalist API Species data, range maps, taxonomy None (read), OAuth (write) Free For /travel/field-guide. Search by taxon name for species info, photos, observation data.
MusicBrainz API Artist/album metadata, release dates None Free Open music database. Rate limit: 1 req/sec. Good supplement to Spotify for J-pop/Mandopop artist data.
Open Exchange Rates Historical exchange rates API key Free tier (1000/mo) More reliable than frankfurter for historical data. Use for trip budget calculator currency conversion.
API key security: All keys go in .env.local and are only accessed in Server Components, Route Handlers (/api/*), or next.config.js server-side runtime. Never in NEXT_PUBLIC_* unless the API explicitly allows public exposure (frankfurter, NUSMods, Jikan do — they're keyless).

Content Flywheel

A repeatable rhythm. The site stays alive without requiring daily output.

W
Weekly (20–30 min each)
▼
1 devlog — what you worked on in Liveling or any project
1 market note — one chart + one observation, no predictions
1 entertainment rec — anime ep, song, game moment
Update /about/now if anything changed
M
Monthly
▼
1 deep project case study or devlog series
1 travel or field guide page
1 new tool or meaningful tool update
Dew Dispatch newsletter issue
Q
Quarterly
▼
Site roadmap update — what you're planning next
Personal review — what you built, learned, explored
Best discoveries — books, tools, anime, music
Transparency update — traffic, costs, what's working
The key insight: A devlog, a market note, and an entertainment rec are all just quick observations written down. At this cadence the site always feels alive without you burning out.
Cross-posting: Every piece of writing also lives in its vertical. A Japan devlog lives in /travel/japan/blog AND /writing/travelogues. One piece of content, two canonical homes.
